API 可見性的類型

四種API可見性類型

##解決方案內部API (Solution-Internal API)

解決方案內部API是用於應用程式內部的元件之間的通信，僅限於應用程式內部使用，不對外開放。

  • 用途：將應用程式依邏輯分解成元件，這些元件暴露API來進行內部通信。
  • 特點：
    • 僅限於內部系統使用，通常由單一專案或產品負責開發、營運和維護。
    • 主要用於服務內部元件的協同工作。

例子：

  • 後端服務間的API：例如，處理用戶資料的服務與支付服務之間的API。
  • 內部數據庫管理API：例如，內部API負責處理數據存取和管理。

夥伴API (Partner API)

夥伴API是供特定合作夥伴使用的API，通常會有某些存取限制，且需要合作夥伴進行認證。

  • 用途：用於兩個或多個組織之間進行系統集成和數據共享。
  • 特點：
    • 主要用於與合作夥伴或外部系統的集成。
    • 需要身份驗證和授權來確保安全。
    • 通常用於跨企業或跨部門的協作。

例子：

  • 支付API：例如，支付平台提供的API給商家用於支付處理。
  • 第三方物流API：例如，電子商務平台與物流公司之間的API集成。

公開API (Public API)

公開API是對所有開發者開放的API，任何人都可以使用，通常不需要身份驗證。

  • 用途：促進開發者生態系統的發展，讓開發者可以創建與平台或服務集成的應用程式。
  • 特點：
    • 完全對外開放，任何開發者都可以使用。
    • 通常會有一些存取限制（如API密鑰、速率限制）。
    • 促進平台或服務的生態系統發展。

例子：

  • Google Maps API：用於集成地圖和位置服務。
  • Twitter API：用於與Twitter平台進行數據交互。

社群API (Community API)

社群API是為了特定的社群或開發者群體提供的API，通常會有特定的授權和存取條件。

  • 用途：促進特定社群或開發者群體之間的協作，通常是為了推動開放源代碼或某個特定領域的開發。
  • 特點：
    • 用於特定社群或開發者群體，可能需要加入特定的開發者計劃或社群。
    • 可能會有更高的技術支持和專業資源。
    • 提供開放源代碼或協作平台，推動技術共享。

例子：

  • GitHub API：開放給開發者用於操作GitHub上的資源，促進開源協作。
  • Slack API：提供給開發者用來擴展Slack平台功能的API。
