Web API設計 - 設計與挑戰

發表於2025-01-19|更新於2025-12-14|API
|瀏覽量:

API 設計

全名 Application Programming Interface

透過了解需求的優先級，設計者可以在這些選項之間找到平衡。

1. 通用 API vs 特定 API

所有客戶端是否應該使用同一個通用 API，還是為不同需求提供特定的 API？

  • 通用 API
    • 優勢：
      • 高重用性，減少重複開發。
      • 易於維護，統一版本管理。
    • 劣勢：
      • 可能過於複雜，導致學習曲線陡峭。
      • 難以滿足特定需求，客戶端需進行額外的資料處理。
      • 設計過於通用時，API定義模糊不清。
    • 例子：GraphQL 允許客戶端自行定義所需的資料結構。
  • 特定 API
    • 優勢：
      • 精準滿足需求，減少客戶端資料處理負擔。
      • 更易於理解，適合特定場景。
    • 劣勢：
      • 重複開發，增加維護成本。
      • 客戶端需求變更時，需頻繁修改 API。
    • 例子：REST API 中 /users/{id} 提供特定使用者的資訊，而 /users 提供所有使用者的資訊。

2. 細 vs 粗的EndPoint(端點)與操作範圍

如何在API的功能廣度與深度之間取得平衡?

  • 細粒度端點
    • 優勢：
      • 更靈活，可精確執行操作。
      • 客戶端只需請求所需的資源。
    • 劣勢：
      • 需要多次請求，可能影響效能。
      • 增加整體 API 的數量和複雜度。
    • 例子細粒度：/users/{id}/profile 和 /users/{id}/settings 提供不同的資料。
  • 粗粒度端點
    • 優勢：
      • 一次請求攜帶更多資料，減少網路延遲。
      • 簡化 API 結構。
    • 劣勢：
      • 客戶端可能獲取多餘資料，浪費頻寬。
      • 無法精細控制資源。
    • 例子粗粒度：/users/{id} 一次返回完整的使用者資訊，包括個人資料與設定。

3. 少操作攜帶多資料 vs 多操作攜帶少資料

偏向於單一操作返回更多資料，還是分多次操作返回精簡資料？

  • 少操作攜帶多資料
    • 優勢：
      • 減少請求次數，提升效能。
      • 適合高延遲環境（如行動網路）。
    • 劣勢：
      • 資料冗餘，可能浪費頻寬。
      • 客戶端可能需要處理多餘資料。
    • 例子少操作攜帶多資料：/dashboard 返回所有儀表板資料，包括統計數據和圖表。
  • 多操作攜帶少資料
    • 優勢：
      • 精準控制，避免多餘資料。
      • 更適合動態需求變化的場景。
    • 劣勢：
      • 請求次數多，可能影響效能。
    • 例子多操作攜帶少資料：/dashboard/stats 和 /dashboard/charts 分別返回統計數據和圖表。

4. 資料現時性 vs 正確性

是否應該分享舊資料，還是等待正確資料的更新？

  • 現時性優先
    • 優勢：
      • 快速回應，提高使用者體驗。
      • 適合對延遲敏感的場景（如即時聊天）。
    • 劣勢：
      • 資料可能不完全正確。
    • 例子現時性：即時推送股票價格，但可能延遲數秒。
  • 正確性優先
    • 優勢：
      • 確保資料一致性，避免誤導使用者。
      • 適合需要高準確度的場景（如財務系統）。
    • 劣勢：
      • 回應時間可能較長。
    • 例子正確性：等待交易結算後再更新最終價格。

5. 穩定規約 vs 快速改變規約

如何在保持相容性的同時進行 API 的迭代？

  • 穩定規約
    • 優勢：
      • 減少對現有客戶端的影響。
      • 提高使用者信任。
    • 劣勢：
      • API 演進速度慢，可能限制新功能的引入。
    • 例子穩定規約：使用版本控制（如 /v1/users 和 /v2/users）來引入新功能。
  • 快速改變規約
    • 優勢：
      • 能夠快速適應需求變化。
      • 更靈活地引入新功能。
    • 劣勢：
      • 可能導致版本碎片化，增加維護成本。
    • 例子快速改變規約：使用非破壞性變更（如新增欄位）來保持向後相容。

API 設計與考量

開發者體驗 (DX)

API 設計影響開發者的學習與使用體驗，並對系統效能、安全性和可管理性產生重大影響。後端在開發API時應考量前端接API角度。

💡DX = 開發者體驗，體驗部分: 功能性、穩定性、易用性、清晰性。好的API具備這些特質。

考量

提供一個 RESTful API，支持標準的 CRUD 操作，並附上詳細的 Swagger 文件，方便開發者快速上手。

  • 提供清晰的文件與使用示例，減少學習曲線與反覆溝通。
  • 設計直觀的 API 接口，讓開發者易於理解和使用。

客戶端多樣性

不同客戶端的需求多樣且持續變化，API 必須在統一性與靈活性之間取得平衡。可以設計通用API 為多個前端應用提供單一 API，讓 Web 和移動應用可以請求各自需要的資料結構。

考量

  • 建立通用 API 並提供可擴展的自定義功能。
  • 使用 GraphQL 技術，讓客戶端根據需求自定義請求的資料。

動態市場

API 提供者間的競爭導致頻繁改變，客戶端可能無法接受；同時，客戶端希望標準化以避免被鎖定。

這邊舉Google例子:

Google Maps 提供穩定的 API 版本控制（如 v1, v2），同時支援舊版 API 一段時間，方便客戶端逐步過渡。

考量

  • 制定 API 標準，促進兼容性與可移植性。
  • 提供清晰的版本控制策略，避免頻繁不相容改變。

分散式系統挑戰

不穩定網路可能影響 API 的可用性與回應時間，提供高服務品質（QoS）保證困難。可以使用重試與快取策略做改善

例子:

API Gateway 中加入重試策略，當後端服務短暫不可用時，自動重試請求。

考量

  • 實現穩健的錯誤處理與重試機制。
  • 使用快取與負載均衡技術提升效能。

控制權問題

開放 API 會導致失去部分控制權，可能承受外部壓力。因此需要限制 API 的請求速率（Rate Limiting）以防止濫用，同時通過 API 金鑰控制訪問權限。

考量

  • 清楚定義 API 的範圍與使用限制。
  • 定期審查 API 暴露的功能與資料，避免過度暴露。
