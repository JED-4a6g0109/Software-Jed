基礎架構即程式碼(IaC)

基礎建設即程式碼（Infrastructure as Code, IaC） 是一種強化 DevOps持續交付軟體品質與版本控制的技術，透過描述性的模型來定義與部署基礎建設資源，例如以下：

  • net
  • vm
  • api gateway
  • cdn
  • serverless fcuntion

IaC把煩雜的設置抽象出來，且用讓人看得懂的設置使得部署更好維護，以及快速拓展應用程式無需一一設置，直接實際舉個例子：

需求如下
建立一個簡單的 HTTP API，當訪問 /greet/{name} 時返回一個問候語。
使用 AWS Lambda 提供業務邏輯，並通過 API Gateway 提供 HTTP 端點。

首先需要寫個python lambda

接著做

  1. 手動創建 Lambda
    在 AWS Management Console 中進入 Lambda 控制台，點擊 “Create Function”。
    填寫函數名稱、運行時（Python 3.9）、內存配置等。
    上傳代碼文件。

  2. 手動配置 API Gateway
    在 API Gateway 控制台中創建 REST API。
    新增資源 /greet/{name}。
    配置方法為 GET，並連接到剛創建的 Lambda 函數。

  3. 測試與部署
    手動部署 API 到一個 Stage，例如 dev。
    記錄 API Gateway URL 以供測試。

以傳統部署方式會在每個步驟都做一次，其實隱藏了龐大的技術債，試著思考現在需求看戲簡單，但如果未來要連DB或者設置VPC以及Load Balancer等複雜設置就會引起以下問題：

  • 沒有詳細文件無從設置

  • 設置的UI改變了，需要從新學習如何設置

  • 需要建立多個新服務每個都要做重複事情浪費時間

  • 設置上可能人為手動設置錯誤導致不一致

  • 維護上困難，可能是技術人員漏更新，或者難以理解導致

以上任何一點都是隕石級風險，其中一步有問題就沒辦法部署了，為了要解決這些問題就是IaC核心的概念，我們想看改成IaC會多沒完美

使用serverless.yaml設置

接著sls deploy馬上結束，非常快速

IaC 的優勢

  • 自動化部署：只需撰寫配置文件，運行命令即可完成整個基礎設施的設置和代碼部署。

  • 快速拓展：只需新增配置即可添加新功能，例如多個 Lambda 函數或 API 路徑。

  • 可重現性：同樣的 serverless.yaml 每次部署都能生成一致的環境。

  • 維護性佳：抽象個多煩雜設置，可讀性佳

AWS、GCP、Azure，大部分Iac都支援YAML、JSON 和 XML，例如AWS Serverless 就使用YAML來進行AWS部署與設置

IaC 與手動配置的差異

比較項目 使用 IaC 手動配置
部署速度 一條命令完成部署，幾分鐘內即可完成 需要進入多個控制台，逐步手動設置，每次操作需耗時 10-30 分鐘
可重現性 配置文件記錄完整環境，隨時可以重複部署相同環境 每次手動操作可能產生不同配置，容易出現錯誤
維護成本 配置集中在代碼中，統一管理和版本控制 需要記住每次手動配置，排查錯誤較為困難
自動化 支持自動化測試與 CI/CD 管道集成 很難與自動化流程整合
擴展性 添加新功能只需更新代碼和配置，重新部署即可 每次擴展功能需要重新進行大量手動操作
