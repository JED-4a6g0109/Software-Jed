防禦繞過 (Bypass) 分析

有些後端工程師會寫一段邏輯來「移除」危險字串，例如把 ../ replace 成空字串。但事情真的有這麼簡單嗎？

案例一：利用「單次處理」缺陷

這代表把字串中的 ../ 換成空字串。

當你輸入 ....// 時，程式的處理過程如下：

  1. 原始輸入： ....//
  2. 偵測危險： 程式發現中間有一個 ../
  3. 執行刪除： 程式把那個 ../ 拿掉
  4. 最終結果： 剩下的前後字元自動黏在一起，變成了 ../

案例二：Double Encoding

除了利用後端程式處理漏洞外，也能用 Double Encoding 的方式攻擊。

這個 payload 的核心在於 %252f 是如何一步步變回 / 的：

  1. 原始字串： ..%252f
  2. 第一次解碼，通常由 Web 伺服器或 WAF 執行：
    • %25 是百分比符號 % 的編碼
    • 解碼後，%25 變回 %，後面的 2f 跟著湊上來
    • 結果變成：..%2f
    • 此時過濾器或 WAF 檢查時，..%2f 看起來不像危險的 ../，因此放行
  3. 第二次解碼，通常由後端應用程式或資料庫執行：
    • %2f 被解碼成 /
    • 結果變成：../

路徑驗證

有些工程師為了安全，會寫一個檢查規則：「這個請求的開頭一定要是 /var/www/images/，如果不是，就報錯。」

這種做法是為了確保使用者只能讀取圖片資料夾裡的東西。但如果程式只是檢查開頭，卻沒有正規化與驗證最終路徑，攻擊者就可以先給它想要的開頭，再用 ../ 爬出去。

系統是這樣看的：

  1. 滿足檢查條件： 程式檢查字串開頭是否為 /var/www/images/，答案是「是」，因此通過
  2. 路徑解析：
    • 進入 /var/www/images/../，回到 /var/www/
    • 進入 /var/www/../，回到 /var/
    • 進入 /var/../，回到 / 根目錄
  3. 最終執行： 程式雖然以為它在處理圖片目錄下的檔案，但實際上作業系統被引導去讀取 /etc/passwd

防範方法

主控權由後端處理，前端透過 ID 查詢

前端只傳遞 ID 給後端，由後端進行檔案查找與比對。這樣也能處理檔案命名相同與隱私問題，此方案最推薦。

權限設置

明確設定伺服器上的 role 與檔案權限，不要設定 777 權限，也不要使用 root 執行服務。如果使用雲服務，也請設置好 IAM、bucket policy、security group 等權限。

監控防禦

可透過監控框架針對 log 進行可疑請求篩選。當發現可疑請求時，可以立即禁止該 IP 訪問或發送預警。

後端 value 檢查 / 後端參數檢查

request 進來的參數都要做驗證，避免被 bypass，提升安全性。

建議做法包含：

  • 使用 allowlist，只允許合法檔名或合法 ID
  • 對路徑做 canonicalization，確認解析後的最終路徑仍在允許目錄內
  • 禁止使用者直接控制完整檔案路徑
  • 避免單純使用字串 replace 作為唯一防線
  • 搭配最小權限原則，降低漏洞被利用時的影響範圍
