JWT 不是無狀態嗎？前後端分離下，如何處理「User登出」的 Token 管理難題？

發表於2026-05-18|更新於2026-05-27|APIJWT
|瀏覽量:

科技趣事

美國矽谷人形機器人新創公司 Figure AI 發起了一場長達 10 小時的人類與機器人物流撿貨大戰，並在 YouTube 上進行直播，最終人類獲勝了。但對於身體上，如手起水泡和疲勞，雖然機器人 BOB 輸了 200 多件貨品，但她還能為老闆繼續賺錢。

JWT 不是無狀態嗎？前後端分離下，如何處理「User登出」的 Token 管理難題？

JWT 是無狀態、擴展性佳，主要用於在網路應用程式之間安全地傳遞資訊，適合分散式系統及單一登入（SSO）。用途是處理身分驗證（Authentication）與授權。

前端會去打 POST /login API 取得 access_token。但這裡有個問題，每當 access_token 過期時，User 都要重新打帳號密碼透過 /login 來拿到 access_token。

通常 access_token 的時效會設置 5~30 分鐘，依系統可接受程度設計。access_token 放在 User 端會有被取得風險。為了避免 Token 外洩後一直被使用，這邊是以便利性和安全性做出權衡：安全 > 便利。

但如果 access_token 設很短，假設為 15 分鐘，那 User 每 15 分鐘被強制登出，還要打一次帳號密碼才能登入，豈不是很怪嗎？而且 User 打帳號密碼這件事情其實也是有風險的。

那該怎麼解決這問題呢？這時就需要 Refresh Token。/login Response 會回傳 refresh_token，refresh_token 時效通常更久，1 天至 30 天都有。

Response：

後端會再開一個 POST /refresh API，前端會在背景默默幫使用者替換。當前端訪問後端 API 時發現 401，會自動打 /refresh API 來取得新的 access_token。

Response：

以上介紹了 JWT Token 登入的情境與解釋。現在要來考慮一個情境：登出時，前端清空了 access_token，但是 access_token 本身還是有效，取得到 token 的人仍然有權限打後端 API。

前端用戶登出時，該如何管理 Token 或者時效？以下有幾種方式。

黑名單模式

需要一個管理 token 的資料庫，這邊以 Redis 來舉例。登出時把 Token 加進 Redis 黑名單紀錄，每次請求時比對，就能對登出的 access_token 進行管控。Key 可以使用 jti 保證唯一，並計算時間設置 TTL。

graph TD
    A[User 點擊登出] --> B[前端發送 Logout 請求]
    B --> C{伺服器解析 JWT}
    C -- 有效 Token --> D[提取 Payload 中的 jti]
    C -- 無效/過期 --> E[回傳 401/403]
    D --> F[計算 Token 剩餘存活時間 TTL]
    F --> G[將 jti 存入 Redis]
    G --> H[設定 Redis Key 過期時間 = TTL]
    H --> I[回傳登出成功]

設計權衡

✅ 實作簡單，效果直接
✅ Redis TTL 自動清理，不會資料無限膨脹
❌ 每次請求多一次 Token 資料庫查詢
❌ 多一個快取資料庫或者 Table 維護

短時效 access_token + 登出撤銷 refresh_token

將 access_token 時效設置 5 分鐘至 15 分鐘，而 token 資料庫 table 欄位需要定義 revoked 欄位來處理。

設計權衡

✅ Access Token 短效，風險時間縮小
✅ Refresh Token 可及時處理盜用情境
✅ 支援所有裝置登出，可實現撤銷該 user 的所有 Refresh Token
❌ 需多維護 table 與撤銷邏輯
❌ Access Token 5~15 分鐘內仍有效，金融盜刷這種場景應該不能容許這種時效

新增 Token 版本核對

在既有的 User table 上多一個 token_version 欄位，並把 token_version 寫進 payload。每次呼叫後端 API 時，在 verify 流程多一段 token_version 驗證，判斷是否為有效 access_token。

這方法有個缺點是要針對特定裝置撤銷 Token 無法做到，因為 User table 的 token_version 欄位無法分區裝置，所有裝置都共用此欄位，因此無法實現單一裝置撤銷。

要馬全部撤銷，無法實現單一裝置。這情境或許會出現在現實情境，例如手機遺失就須全部撤銷 Token，無法只撤銷單一手機。

設計權衡

✅ 實作極簡，不需要額外表或快取資料庫管理
✅ 即時撤銷 Token，且是全裝置
❌ 每次請求仍需查一次 DB
❌ 無法指定單一裝置登出與撤銷

如果不想多一個 Redis，或者不想在既有資料庫上開一個 table 來管理 token，那此方法或許是不錯的選擇。

總結

沒有絕對完美的答案，只有最適合當下業務場景的權衡（Trade-offs）。當系統規模夠龐大、業務邏輯更複雜時，實務上也常常會將這三種方法進行組合設計，以達到便利性與安全性的最高平衡。
