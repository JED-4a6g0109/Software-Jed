JWT 在微服務場景是很常見的，JWT 通常由三個部分組成，以點（.）隔開：

  • Header（標頭）：宣告標記類型（JWT）與使用的加密演算法，如 HS256、RS256。
  • Payload（聲明/內容）：存放使用者資訊，如 username: "wiener"、role: "user"。
  • Signature（簽名）：伺服器用私鑰或金鑰對前兩部分進行加密產生的簽名，用來防止內容被竄改。

本篇將探討 JWT 在開發上有哪些風險需要考量，以及從攻擊者角度理解如何透過 JWT 攻擊服務，並提供解決方向。

1. 透過未驗證簽名繞過 JWT 身分驗證

只 Decode 而非 Verify。當使用者發送請求時，後端應該要先驗證 Signature 是否正確，確認沒有被改過，才能信任 Payload 裡的資料。

但在有些測試環境，或者開發者毫無資安意識時，伺服器根本沒有去驗證 JWT 的簽名部分，就直接信任 Payload 裡面的宣告。許多 JWT 函式庫提供了兩種方法：

  • decode()：單純把 Base64 字串轉回明文，不驗證 Signature。
  • verify()：先用密鑰驗證 Signature，通過後才解析內容。

開發者可能誤用了 decode()，或者在寫測試程式碼時忘記切換回生產環境的 verify()。

例子：建立 FastAPI 後端服務。

攻擊手段：在前端隨便造一個 JWT，例如 Payload 寫 {"sub": "administrator", "role": "admin"}，第三部分簽名隨便亂填甚至不填，直接發送請求，這個 API 就會誤以為他是管理員。

2. 透過有缺陷的簽名驗證繞過 JWT 身分驗證

與前面的「完全不驗證」不同，這一次後端確實有執行驗證程序，但因為邏輯設計上的低級失誤，造成驗證缺陷。

正常情況下，健全的 JWT 驗證邏輯是：

如果請求帶有 JWT，就必須用金鑰驗證它。驗證失敗就拒絕。

然而，不小心把驗證邏輯寫成選擇性驗證，只在某種特定條件下才去驗證簽名。

例如後端的邏輯是：「如果 Token 結構裡有簽名，我才驗證；如果 Token 裡面根本沒有簽名，只有 Header 和 Payload，我就直接放行。」

3. 弱金鑰（Weak Secret Key）

在使用對稱加密演算法如 HS256 時，伺服器簽名與驗證使用的是同一把金鑰。

安全性前提：這把金鑰必須非常複雜，使用隨機亂數且長度足夠，並且只有伺服器內部知道。如果開發者貪圖方便，使用了常見的單字如 secret、password123、flag，或框架預設值如 django-insecure-...，攻擊者就可以透過離線暴力破解出來，並簽章出一個惡意 Token。

4. JWK Header 注入攻擊

伺服器用自己的私鑰簽名，驗證時應該從受信任的來源取得公鑰。但如果後端盲目信任 JWT Header 裡的公鑰資訊，並直接拿該公鑰去驗證簽名，就可能出現問題。

    B --> C[2. 修改 Payload 為 administrator]
    C --> D[3. 用自己的私鑰對 Token 進行簽名]
    D --> E[4. 將自己的公鑰注入到 Header 的 jwk 欄位中]
    E --> F[5. 發送 Token 給伺服器]
    F --> G{後端讀取 Token 內的 jwk 公鑰}
    G -->|用駭客的公鑰驗證駭客的私鑰簽名| H[驗證成功，誤信身分]

另一種 JWT Header 支援一個選用參數叫做 jku（JSON Web Key Set URL）。它是一個網址，用來告訴後端：

如果你要驗證我這個 Token，請去 jku 指定的網址下載公鑰清單（JWKS），裡面有可以驗證我的公鑰。

後端程式碼在收到 Token 後，真的乖乖聽從 Token 的指示，發送 HTTP 請求去下載該網址的公鑰來做驗證，而沒有檢查這個網址到底是不是自家的伺服器。

實務上較少看到這種驗證方式。後端永遠都不要信任前端的任何請求。

結論

JWT 後端在設計上要謹慎思考整體流程是否有缺陷，後端也要做好完整的驗證和檢查，不要盲目相信任何請求。
