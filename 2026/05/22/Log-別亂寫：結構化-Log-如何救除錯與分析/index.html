Log 別亂寫：結構化 Log 如何救除錯與分析

發表於2026-05-22|更新於2026-05-27|軟體開發後端
|瀏覽量:

軟體工程師在開發時逃不掉寫 Log 的情況，而 Log 級別分為：

級別 (Level) 說明 (Description) 使用場景
DEBUG 詳細的除錯資訊 開發期間追蹤變數、邏輯流程。
INFO 一般狀態資訊 紀錄程式正常運行的關鍵節點，如啟動、關閉。
WARNING 警告資訊 出現不尋常但暫不影響運行的狀況，如 API 廢棄。
ERROR 一般錯誤 發生問題導致部分功能失效，如讀檔失敗。
CRITICAL 嚴重錯誤 極端嚴重的錯誤，程式通常難以維持運作。

基本上不太會使用 DEBUG 這個級別，一方面測試用放進程式碼會很亂，另一方面這個級別沒控制好，Log 搜尋會浪費儲存空間與效能。大多數 package 都會有 debug 級別的詳細資訊顯示。

因此在工作中常用的剩 INFO、WARNING、ERROR、CRITICAL，以下介紹一下使用場景。

INFO

紀錄詳細 API 請求資訊，方便分析用量或行為等情境，例如機密系統的 Audit。

WARNING

在後端開發最常見用在廢棄的 API。有些時候因為歷史因素 API 開放出去，但業務沒辦法追溯情況，就會先暫時寫 Log 來觀察還有誰在用。

另外，網路不穩造成短暫的超時也會寫。這種預期內的問題且須追蹤，我自己就會使用 WARNING。

ERROR

重大問題或者不允許出現的問題，像是寫 DB 時失敗，或者 API 非預期錯誤放在 try except 裡面來追溯。非預期或者不允許錯誤都會下 ERROR。

CRITICAL

其實我工作上也只用到 WARNING 與 ERROR，沒有實際用 CRITICAL。但這級別屬於重大事件、需要立即處理級別的。

我自己思考想像是主功能掛掉了，或者是重要 API 被濫用或滲透，又或者資料被刪除等緊急事件。

以上為級別的使用情境與定義。接著來講最重要的環節：Log 要怎麼寫？

這邊我強烈推薦一定要結構化！結構化！結構化！

最隨便的 Log 就是寫一段字串，接著來解釋為何不應該亂寫，以及該如何寫。

可讀性

單體服務架構寫字串隨便寫，隨便開心。但如果是微服務出大事看 Log，你絕對 OOXX。

因為 Log 本身不管是伺服器、Gateway，或者其他 middleware 都會記錄 Log，在看 Log 時會很痛苦。

好分析

各位開發一定有遇到，有些時候資料不想存在 DB，但就想記錄如 audit 或者 API Request 時就會寫進 Log。

而這些資料未來老闆需要時，如果是寫成字串方式存儲就會很想死，ETL 的 E 就會非常麻煩。

但如果將 Log 結構化，你只需要將 JSON String 轉換成 JSON，就能無痛處理又舒服。而且每次寫 Log 增加的 key 都可能會增加，因此可以在 JSON 新增 version，未來在處理資料可依據 version 去 mapping Log 的資料結構，也對於後續 Extract 更便利，更好分析資料。

好除錯

如果結構化，就能針對 JSON 某個 key 進行快速篩選與查找，精準搜尋與處理。

結構化 Log 基本屬性

那結構化該如何寫呢？又有哪些屬性呢？以下條列最基本屬性：

  • 時間（UTC）
  • 伺服器名稱
  • Log 級別
  • Log 訊息

這幾個屬性為必要，其餘就根據自己需求來新增 key。像是服務如果不想用框架做鏈路追蹤，可以自己添加 trace_id 來整併。

總結

好的 Log 級別不僅幫助工程師排查，也能明確當警示指標，而結構化 Log 對於分析和除錯都是好的方式。
