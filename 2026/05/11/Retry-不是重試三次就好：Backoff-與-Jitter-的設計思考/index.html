後端應用服務重試機制 (Retry) 設計筆記

在軟體開發中，重試機制 Retry 是很常用的設計之一。在後端服務開發中，除了自身服務 API 外，也有可能連到外部服務做操作，這些場景都可能會需要設計重試機制。

以上實現完成！太好了，準時下班！如果是這樣想，那肯定要加班了。首先要討論的是：什麼時候該用重試機制？

重試適用於 突發性、發生率低、不頻繁 的場景，例如：

情境 說明
網路不穩定的偶發性 如掉包、瞬間斷線
服務突發忙碌導致 API 超時 暫時性負載過高

⚠️ 關鍵前提：冪等性 (Idempotence)

冪等性是指操作或 API 無論執行一次或多次，結果皆相同，且不會產生副作用，例如重複扣款、重複建立數據。

特殊場景補充：請求 API 建立資料成功，並在資料庫 insert 資料，但 Response 時網路掉包，可能引發重試。因此冪等性是很重要的考量，除非被請求的 API 有針對這種場景做特殊處理。

驚群問題 (Thundering Herd)：整齊劃一的致命同步

假設在有規模的系統下，A 服務不一定是單一的，它可能是動態擴容，或者固定佈署好幾個服務。

試想一下，如果有 100 個 User 使用 A 服務，而 A 服務呼叫 B 服務，但 B 服務發生突發錯誤，導致 A 服務開始重試。這等於 100 個平行請求在約幾百毫秒內，由 A 服務器重複瘋狂打 B 服務。那 B 服務器的負載肯定會越來越大，最終可能被打掛。

如果是流量小的服務，這種方式還可以接受；但在有一定流量的服務中，就不建議這樣做。

為了解決這問題，需要引入 Jitter (隨機抖動)。不應該都固定 3 秒，而是給它們一個隨機範圍：

  • 有人等 2.8 秒
  • 有人等 3.2 秒
  • 有人等 3.5 秒

這樣，100 個請求就會在時間軸上被「攤平」，B 服務接收到的壓力會從 「尖峰」 變成 「緩丘」。

重試不只有在應用層，也可以在這些地方處理：

位置 負責角色 優點 缺點 適用場景
應用程式層 (Application) 開發者 邏輯最精確，可根據業務代碼，例如 404 不重試、500 才重試，做細緻判斷。 程式碼冗餘、維護成本高，開發者需自行處理退避策略與 Jitter。 呼叫第三方 API、處理複雜業務邏輯判斷時。
基礎設施層 (Service Mesh) 維運 / DevOps 語言無關性、開發不需介入代碼，可由 Service Mesh，例如 Istio，統一管理重試政策。 架構複雜度提高，難以處理深層的業務邏輯錯誤。 內部微服務之間的大規模通訊、需要統一管理流量時。
API 閘道層 (API Gateway) 系統架構師 攔截外部錯誤、統一流量入口，保護後端服務免於短暫波動影響。 策略較單一，缺乏業務細節。 外部請求進入系統的第一道防線、簡單的超時重試。
訊息佇列層 (Message Queue) 非同步架構師 具備強大的緩衝能力與韌性，適合處理最終一致性的任務，不會遺失資料。 延遲較高，且處理邏輯必須嚴格遵守冪等性。 背景任務、非即時性的數據同步、不容許失敗的操作。

設計重試機制時，需注意以下幾點：

要素 說明
重試次數 設定合理的上限，避免無限重試
等待時間 加入退避策略，固定、線性或指數
隨機抖動 (Jitter) 打散同步重試，避免驚群問題
冪等性 確保重試不會造成副作用
設計策略 根據服務流量規模選擇適合的退避算法

💡 核心原則：重試機制的目的是幫助系統恢復與避免資料因特殊狀況而遺失，而不是增加伺服器負擔。
