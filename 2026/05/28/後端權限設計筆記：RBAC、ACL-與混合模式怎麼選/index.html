後端權限設計筆記：RBAC、ACL 與混合模式怎麼選？

後端在帳號、API、組織、後台管理、多租戶系統等場景中，常常都會遇到一個核心問題：

誰可以對什麼資源做什麼事？

這就是權限設計要處理的事情。

本文會介紹最常見的兩種權限模型：

  • RBAC：Role-Based Access Control，角色型存取控制
  • ACL：Access Control List，存取控制清單

我會用「高級社區帝爆大樓」作為日常例子來說明，快速理解兩者差異、優缺點，以及實務上為什麼常常會使用 RBAC + ACL 混合模式。

RBAC 的核心概念是：

系統不直接把權限發給個人，而是先把權限發給角色，再把角色指派給使用者。

RBAC 的好處是可以用角色簡化權限控制。例如「一般住戶」這個角色被賦予健身房使用權後，所有一般住戶都能繼承這份權限，不需要逐一幫每個人設定。

但 RBAC 也有一些問題：當業務狀態越來越複雜，角色與權限可能會不斷膨脹。如果硬用 RBAC 去模擬 ACL 的細粒度行為，維護成本會變得很高。

RBAC 的常見問題

情境一：角色特殊授權來回調整，增加風險

假設住戶帶訪客臨時使用健身房，每次住戶帶訪客時，警衛都需要針對訪客開放健身房權限。

  • 有人忘記改回來
  • 臨時角色越建越多
  • 權限異動變得難以追蹤
  • 最終衍生維護成本與資安風險

換成軟體系統來看，就是用 RBAC 模擬 ACL。少量還好，但太頻繁就會讓角色設計變得很混亂。

情境二：角色對複雜條件難以維護

假設健身房人數過多，社區改成分時制度。因為帝爆大樓太熱門，住戶有 2000 人，但健身房只能容納 50 人，所以被切成四個時段：

  • 06:00 - 10:00
  • 10:00 - 14:00
  • 14:00 - 18:00
  • 18:00 - 22:00

如果硬用 RBAC 表達，可能會拆出：

  • 一般住戶_早上時段
  • 一般住戶_中午時段
  • 一般住戶_下午時段
  • 一般住戶_晚上時段

看起來還可以接受，但如果條件再加上棟別、樓層、會員等級、特殊活動日，角色數量會快速爆炸。

ACL — Access Control List（存取控制清單）

ACL 的核心概念是：

直接在資源上定義「誰」可以做什麼。每個資源各自維護一份清單。

ACL 對細粒度權限非常靈活，但它也有明顯缺點。

ACL 的常見問題

情境一：重複定義問題

假設帝爆大樓有 2000 位住戶。如果所有資源都使用 ACL 管理，你可能需要在健身房、停車場、公共空間等每個資源上都塞入一大堆使用者清單。

這代表 ACL 如果用在大量共同權限上，會造成資料重複與維護負擔。

情境二：大量 CRUD 更新問題

如果突然有 300 位用戶搬進來，或 500 位用戶搬出去，你就需要逐一修改每個資源上的 ACL 清單。

這種大量異動很容易變成高風險操作：

  • 更新不完整
  • 資料殘留
  • 權限沒有確實撤銷
  • 不同資源的 ACL 狀態不一致

所以 ACL 很適合處理細粒度與特例，但不適合拿來管理大量共同權限。

RBAC + ACL 混合模式

RBAC 與 ACL 各自有適合的場景：

  • RBAC 適合處理大多數人的共同權限
  • ACL 適合處理少數人的特殊授權

因此實務上常見的方式是混合使用。

建議的判斷原則

常見原則如下：

  1. 明確拒絕優先於允許
  2. ACL 可用來處理特例、臨時授權、細粒度資源
  3. 沒有命中任何規則時，預設拒絕
  4. RBAC 處理大範圍共同權限，ACL 處理少數例外

RBAC、ACL、混合模式怎麼選？

場景 建議模型 原因
公司後台管理員、一般使用者、客服等固定角色 RBAC 角色穩定，權限共用度高
文件分享、單一資源授權 ACL 適合針對單一資源指定使用者
SaaS 多租戶系統 RBAC + ACL 角色處理基本權限，ACL 處理租戶與資源特例
臨時授權，例如訪客、維修工、外部協作者 ACL 或 RBAC + ACL 需要過期時間與細粒度控制

總結

RBAC 與 ACL 沒有誰絕對比較好，重點是看業務場景。

RBAC 適合處理穩定、共用、大範圍的權限，例如後台角色、組織職位、會員等級。

ACL 適合處理細粒度、臨時、特例型授權，例如文件分享、指定房間進入權、臨時訪客權限。

混合模式則是實務上常見的折衷：

設計權限系統時，最重要的不是追求模型看起來多漂亮，而是要讓它在真實業務變化下仍然好維護、好追蹤、好撤銷，也不容易因為人工操作造成資安風險。

方法沒有絕對正確，只有適不適合當下的業務與系統複雜度。

軟體工程師在開發時逃不掉寫 Log 的情況，而 Log 級別分為： 級別 (Level) 說明 (Description) 使用場景 DEBUG 詳細的除錯資訊 開發期間追蹤變數、邏輯流程。 INFO 一般狀態資訊 紀錄程式正常運行的關鍵節點，如啟動、關閉。 WARNING 警告資訊 出現不尋常但暫不影響運行的狀況，如 API 廢棄。 ERROR 一般錯誤 發生問題導致部分功能失效，如讀檔失敗。 CRITICAL 嚴重錯誤 極端嚴重的錯誤，程式通常難以維持運作。 基本上不太會使用 DEBUG 這個級別，一方面測試用放進程式碼會很亂，另一方面這個級別沒控制好，Log 搜尋會浪費儲存空間與效能。大多數 package 都會有 debug 級別的詳細資訊顯示。 因此在工作中常用的剩 INFO、WARNING、ERROR、CRITICAL，以下介紹一下使用場景。 INFO紀錄詳細 API 請求資訊，方便分析用量或行為等情境，例如機密系統的 Audit。 WARNING在後端開發最常見用在廢棄的 API。有些時候因為歷史因素 API 開放出去，但業務沒辦法追...
